7  Plotting and Processing Raw Data

Biomedical Engineering
Applied Statistics

Visual inspection and signal processing are beyond the scope of this course. Instead, our focus is on selecting the appropriate statistical tests and applying them correctly to draw inferences about the hypotheses under investigation. These tests require collecting information from a set of experimental units (e.g., participants, motor units, muscle fibers, or any other unit of analysis). Throughout this course, we will refer to this information as variables.

Some variables can be obtained by directly asking participants (e.g., age) or through simple measurements (e.g., body mass and height). Others require processing biomedical signals. In this tutorial, and throughout the exercises in this course, we will work with variables computed from postural sway data (CoG raw data; see Chapter 4).

… we will use R to process CoG data. Students will use RStudio to compute five variables related to postural control, all of which will be used throughout this course.


7.1 Important Considerations

Before processing any data, we must ensure that the quality of the raw measurements is sufficiently high—this is essential in any experiment or study. If you are unsure how to assess data quality, consult the scientific literature in your field and critically evaluate the characteristics of your measurements.

In our case, we collected smartphone orientation data while participants stood quietly. To assess the quality of the recorded data, it is useful to consider the following questions:

  • How was the smartphone held during data acquisition?
  • Which reference coordinate system does the smartphone use to report orientation data?
  • Was the smartphone held firmly enough that changes in its orientation primarily reflect body sway rather than hand movements?
  • What is the expected amplitude of body sway under the different test conditions (e.g., eyes open, eyes closed)?
  • What frequency range should these sways exhibit? In other words, what are reasonable durations for the fastest and slowest body sway movements?

Once we have plausible answers to these—and possibly other—questions, we can visually inspect the data for artifacts. This prior understanding helps us recognize whether the recorded signals are consistent with the expected physiological behavior.

We will begin by loading the raw data from a csv file into the R workspace, as described in the Loading CoG data section.

df <- read.csv(file.path(getwd(), "phyphox_data", "standing_eyes_open", "postural_sways.csv"))


7.2 Inspecting Raw Data

We have just loaded the contents of the file "postural_sways.csv" into the R workspace. This file contains three columns:

  • time: time instants at which the smartphone orientation data were recorded;
  • x_cog: CoG coordinate in the frontal plane1 (medio-lateral direction);
  • y_cog: CoG coordinate in the sagittal plane2 (antero-posterior direction).

If the data were collected as described in Chapter 3, x_cog and y_cog should represent body sway in the medio-lateral and antero-posterior directions, respectively.

To visualize the CoG position in the sagittal plane, we can use plot():

plot(df$y_cog)


Helpful Cosmetics

By calling the plot() function, you may have noticed that R automatically labels both axes. The \(y\) axis is labeled according to the variable passed as the second argument (df$y_cog), while the \(x\) axis is labeled Index. After checking that we have 3997 data points using length(df$y_cog), it becomes clear that Index refers to the sample number, ranging from 1 to 3997.

Another feature worth noting is that each data point is represented by a marker—a black circle with a white fill. This default behavior is appropriate because R was originally designed for statistical analyses involving relatively small sets of observations, typically corresponding to the number of experimental units in a study (often fewer than 20). In our case, however, the CoG measurements form a continuous time series, making a line plot without markers a more appropriate representation.

These and many other graphical settings can be customized when calling the plot() function:

plot(df$time, df$y_cog,                          # x and y data respectively
     type = "l",                                 # line type plot
     col = "turquoise4",                         # color of the line
     main = "Stabilogram in the Sagittal Plane", # title text, shown on the top of the plot
     font.main = 2,                              # use bold font for the title text
     xlab = "Time (s)",                          # label text for the X axis
     ylab = "CoG Y Position (cm)", # label text for the Y axis
     bty = "l",                    # show only left and bottom borders
     )

For a complete list of graphical parameters, consult the documentation for the par() function.

The anatomical interpretation of changes in CoG Y Position—that is, positive values correspond to forward displacement—was added using the arrows() and text() functions, which are omitted from the code listing for simplicity.


Plotting CoG X and Y

Stabilograms are graphical representations of postural sway as a function of time. Since body sway is measured in both the antero-posterior and medio-lateral directions, two stabilograms are available: one for the sagittal plane (Figures 7.1 and 7.2) and one for the frontal plane.

There are two common ways to display these stabilograms.


Stacking Stabilograms

One option is to create a separate plot for each stabilogram. Simply calling the plot() function twice would not achieve the desired result because R would create each plot in a separate graphics device (or overwrite the existing plot, depending on the environment). Instead, we would like to display both stabilograms within the same figure window. This can be accomplished using the layout() function.

desired_layout <- matrix(
  data = c(1, 2), 
  nrow = 2, 
  ncol = 1
  ) # where to plot the data

desired_layout  # column vector, indicating where to place plot 1 (top) and 2 (bottom)

layout(mat = desired_layout) # tell R to draw the next 2 plots with this layout

layout.show(2) # show the layout (outlines) of the next two plots

desired_layout is a matrix with one column and two rows, containing the values 1 and 2. These values specify where each plot should be drawn: the first plot occupies the top row and the second plot the bottom row. Since both plots are intended to have the same size, passing only the matrix desired_layout to the layout() function is sufficient.

We can now call plot() twice to draw the stabilograms for the sagittal and frontal planes.

Show code 
layout(mat = desired_layout) # tell R to draw the next 2 plots with this layout

# first plot - CoG Y stabilogram
plot(df$time, df$y_cog, 
  type = "l", 
  col = "turquoise4", 
  xlab = "Time (s)", 
  main = "Stabilograms in the Sagittal and Frontal Planes",
  font.main = 2, 
  ylab = "CoG Y Position (cm)", 
  bty = "l"
  )

# second plot - CoG X stabilogram
plot(df$time, df$x_cog, 
  type = "l", 
  col = "purple4", 
  xlab = "Time (s)", 
  ylab = "CoG X Position (cm)", 
  bty = "l"
  )


Overlaying Stabilograms

An alternative approach is to display both stabilograms in the same graph. In this case, there is no need to define a plotting layout. We first create one plot with the plot() function and then add the second trace using the lines() function3.

Show code 
# first plot - CoG Y stabilogram
plot(df$time, df$y_cog, 
  type = "l", 
  col = "turquoise4", 
  xlab = "Time (s)", 
  main = "Stabilograms in the Sagittal and Frontal Planes",
  font.main = 2, 
  ylab = "CoG Position (cm)", 
  bty = "l", 
  ylim = range(c(df$x_cog,df$y_cog), na.rm = TRUE)
  )

# second plot - CoG X stabilogram
lines(df$time, df$x_cog, 
  type = "l", 
  col = "purple4", 
  bty = "l"
  )

# adding legend to distinguish the two traces
legend(
  "right", 
  inset = 0.02, 
  c("CoG Y","CoG X"), 
  col = c("turquoise4","purple4"), 
  lty = 1
  )

When overlaying plots, the different traces must be clearly distinguished and appropriately labeled. Here, different colors are used for the two signals, and the legend() function identifies which line corresponds to each CoG coordinate.

Unfortunately, the overlaid plot does not provide the intended visualization of postural sway in the sagittal (CoG Y) and frontal (CoG X) planes. The primary reason is that the two signals have substantially different mean values. To display both time series on the same graph, the limits of the \(y\) axis must be set to encompass the full range of both coordinates, namely [-82, 5], using the range() function (i.e., ylim = range(c(df$x_cog,df$y_cog), na.rm = TRUE)). Because the actual postural sway is only a few centimeters, scaling the \(y\) axis to a range of approximately 87 cm compresses both stabilograms, making the sway difficult to appreciate visually.

The mean value of each time series4—CoG Y and CoG X—is typically not of interest. It reflects the average position of the CoG during the trial rather than the magnitude or dynamics of body sway. In the next section, we will begin processing the raw data so that the two signals can be meaningfully overlaid and compared.


7.3 Processing Raw Data

The appropriate procedure for processing raw data depends on both the characteristics of the data and the scientific question being addressed.

In this tutorial, we will work with postural sway data. Although no specific research question is investigated here, this tutorial provides an opportunity to become familiar with the RStudio IDE, which will be used throughout the course to address a variety of research questions.

The processing consists of two main steps:

  • Pre-processing Sway Data
  • Computing the Variables of Interest


Pre-processing Sway Data

Before computing any postural metric, we must ensure that the raw data have been properly prepared. Otherwise, the resulting metrics may have questionable validity. For example, recording duration is known to significantly influence postural metrics computed in both the time and frequency domains [1]. One way to minimize this effect is to standardize the duration of the recordings before computing the desired metrics.

In this tutorial, we will therefore perform two pre-processing procedures: trimming and demeaning.


Trimming Sway Data

From the plots shown earlier, we can see that approximately 40 s of sway data were recorded. The first few seconds are likely to contain transient effects (e.g., those associated with initiating data acquisition).

To standardize the duration of the recordings, we will retain a 30 s processing window. Specifically, we will discard the first 5 s of data and then keep the next \(N\) samples, where \(N = 30 \times sf\) and \(sf\) denotes the sampling frequency. Before doing so, we must determine the actual sampling frequency.

By default, PhyPhox records orientation data at approximately \(100\textrm{Hz}\) unless configured otherwise. Rather than assuming this nominal value, we will estimate the sampling frequency directly from the recorded timestamps.

Show code 
si <- diff(df$time) # sampling intervals: between consecutive samples

plot(si*1000, 
  ylab = "Sampling interval (ms)", 
  xlab = "Sampling interval number (#)", 
  main = "Interval between consecutive samples", 
  font.main = 2, 
  col = "turquoise4"
  )

The intervals between consecutive samples are not perfectly constant, indicating a small amount of sampling jitter. On average, however, these deviations are on the order of \(1\mu\textrm{s}\), which is negligible compared with the duration of the 30 s processing window.

The estimated sampling frequency is slightly lower than \(100\textrm{ Hz}\). Based on Figure 7.6, the average sampling rate is \(1/0.010008=99.920064\textrm{ Hz}\). Consequently, assuming \(sf=100\textrm{ Hz}\) introduces a cumulative timing error of approximately \(+0.8\textrm{ ms}\) per second of processed data. For example, while 100 samples correspond to exactly \(1\textrm{ s}\) at \(100\textrm{ Hz}\), they actually span \(100/99.920064=1.0008\textrm{ s}\) at the measured sampling frequency.

This discrepancy is sufficiently small to be ignored, and throughout this tutorial we will assume a sampling frequency of \(sf=100\textrm{ Hz}\).

sf <- round(1/mean(si)) # mean sampling frequency (100 Hz)
dur <- 30               # duration of the processing window
y_cog <- head(          # 30 s of CoG Y samples
  df$y_cog[df$time>5],
  dur * sf
  ) 
x_cog <- head(          # 30 s of CoG X samples
  df$x_cog[df$time>5], 
  dur * sf
  ) 
time <- seq_len(dur * sf) / sf  # time vector (sf = 100 Hz)

We used the round() function to approximate the estimated sampling frequency to the nearest integer, which in this case is \(100\textrm{ Hz}\)5. We then used the head() function to retain \(3000\) samples (i.e., \(`dur * sf = 30 \times 100 = 3000\)) of both CoG Y and CoG X after the first \(5 \textrm{ s}\) of recording.

For convenience, the CoG coordinates in the two anatomical directions were stored in the variables y_cog and x_cog.


Demeaning Sway Data

Removing the mean from the data is not strictly necessary, but it simplifies both signal processing and visualization.

To demean \(y_\textrm{cog}\) and \(x_\textrm{cog}\), we simply subtract their respective mean values, computed using the mean() function:

y_cog = y_cog - mean(y_cog, na.rm = TRUE) # demeaned CoG Y
x_cog = x_cog - mean(x_cog, na.rm = TRUE) # demeaned CoG X

Notice the argument na.rm = TRUE passed to the mean() function. This ensures that missing values (NA) are ignored when computing the mean. Such values may arise, for example, if the final line of the imported csv file is empty. We can verify this by inspecting the last few observations with the tail() function:

tail(df,5) # the last 5 observations in df

Computing CoG Metrics

As discussed in Chapter 2, numerous postural metrics have been proposed to characterize CoG trajectories [2,3]. In this tutorial, and throughout the course, we will focus on only three of them: the Elliptic Area (\(EA\)), the Standard Deviation (\(\sigma_{x,y}\)), and the Mean Speed (\(v_{x,y}\)).


Preliminaries on the Elliptic Area

In Section 2.2.5.2 (see Equation 2.12), we described the procedure used to compute the Elliptic Area. Briefly, we seek the two principal axes of the ellipse that best fits the CoG trajectory. These axes correspond to the directions along which the CoG coordinates are uncorrelated. Mathematically, this is achieved by finding the reference system that diagonalizes the CoG covariance matrix.

We begin by constructing the Statokinesigram6:

Show code 
plot(x_cog, y_cog, 
  type = "l", 
  col = "turquoise4", 
  asp = 1,
  main = "Statokinesigram", 
  xlab = "Medial-Lateral (cm)", 
  ylab = "Anterior-Posterior (cm)"
  )

abline(h = 0, lty = "dotted") # adding HORIZONTAL line at 0 cm
abline(v = 0, lty = "dotted") # adding VERTICAL line at 0 cm

The abline() function was used to draw horizontal and vertical reference lines. From Figure 7.8, we observe that, over the 30 s recording, the CoG trajectory spans approximately \(3\textrm{ cm}\) in the antero-posterior direction and \(3\textrm{ cm}\) in the medio-lateral direction. We also note that the principal excursion of the CoG follows a direction slightly rotated counterclockwise with respect to the vertical reference axis.

The axes of the ellipse that best fits the Statokinesigram (Figure 7.8) are computed according to Oliveira et al. [4] (see Equation 2.12).


The Covariance Matrix (\(\Sigma\))

We begin by computing the covariance matrix \(\Sigma\) from the CoG coordinates:

xy <- cbind(x_cog, y_cog) # matrix with N rows (sf * dur) and 2 cols
head(xy, 5)               # shows the first 5 rows of the xy matrix
         x_cog      y_cog
[1,] 0.5778717 -0.9954609
[2,] 0.6006753 -0.9810690
[3,] 0.6258725 -0.9732609
[4,] 0.6432517 -0.9697624
[5,] 0.6478979 -0.9688926
Sigma <- cov(xy)          # covariance matrix

We first used the cbind() function to combine the CoG \(x\) and \(y\) coordinates into a matrix with \(N=3000\) rows (equal to sf * dur) and two columns (x_cog and y_cog). We then computed the covariance matrix \(\Sigma\) using the cov() function from the stats package.

Inspecting Sigma, we observe that it is a symmetric \(2\times2\) matrix, as expected:

Sigma   # displays the covariance matrix
           x_cog      y_cog
x_cog  0.6404470 -0.5573457
y_cog -0.5573457  1.1368876

Its main diagonal contains the variances of \(x_\textrm{cog}\) and \(y_\textrm{cog}\), whereas the off-diagonal elements contain the covariance between the two coordinates. Because covariance is symmetric, these two elements are identical7:


The Ellipse Semi Axes

Our objective is to find a new covariance matrix \(\Lambda\) whose off-diagonal elements are both zero. This corresponds to identifying a new reference system (\(x',y'\)) in which the transformed coordinates are uncorrelated. We obtain this reference system by computing the eigenvalues and eigenvectors of \(\Sigma\) with the eigen() function:

xy_prime <- eigen(Sigma)  # computing eigenvalues and eigenvectors of Sigma 
xy_prime
eigen() decomposition
$values
[1] 1.4987882 0.2785463

$vectors
           [,1]       [,2]
[1,] -0.5445925 -0.8387008
[2,]  0.8387008 -0.5445925

xy_prime is a List containing two named elements: values and vectors. The eigenvalues are stored in xy_prime$values in decreasing order, while the corresponding eigenvectors are stored in xy_prime$vectors.

NoteWhat do these values and vectors represent?

The eigenvalues stored in xy_prime$values correspond to the variances of the CoG coordinates in the transformed reference system defined by \(x'\) and \(y'\). Consequently, the covariance matrix in this reference system is \(\Lambda=\mathrm{diag}(\lambda_1^2,\lambda_2^2)\).

The eigenvectors stored in xy_prime$vectors define the directions of the principal axes of this new reference system.

Their geometric interpretation becomes clearer when inspecting the figure below. For convenience, we store the eigenvalues and eigenvectors in separate R variables:

Show code 
lambda <- sqrt(xy_prime$values) # square root of eigenvalues
eigvec <- xy_prime$vectors      # eigenvectors

# SEMI AXES of the ellipse, scalled with 1.96 standard deviation (1.96 lambda)
arrows(0, 0,                # origin of the vector 
       eigvec[1] * lambda[1] * 1.96, # x' coordinate of the first eigenvector
       eigvec[2] * lambda[1] * 1.96, # y' coordinate of the first eigenvector
       length = 0.05, 
       col = "pink", 
       lwd = 8
       ) #

Notice that we take the square root of the eigenvalues using the sqrt() function. This yields the standard deviations (\(\lambda_1\) and \(\lambda_2\)) of the CoG coordinates in the transformed reference system.

The ellipse semi-axes are shown using two different scaling factors: \(1.00\lambda\) and \(1.96\lambda\). Thus, the vectors displayed in Figure 7.9 have lengths equal to one and 1.96 standard deviations, respectively. The chosen scaling factor determines the size of the ellipse and, consequently, the proportion of CoG samples enclosed by it. For comparison, approximately 34% of observations project within \(\pm1.00\lambda\) along each principal axis, whereas approximately 47.5% project within \(\pm1.96\lambda\) along each axis individually.

NoteHow many CoG samples are included in the ellipse?

Using a scaling factor of \(1.96\), approximately 95% of the CoG samples project within the limits of each principal axis.

The resulting ellipse encloses approximately 85% of the CoG samples (see [4]).


Drawing the Ellipse

Drawing the ellipse is not necessary to compute the elliptic area, since this calculation only requires the eigenvalues (i.e., \(\Lambda=\mathrm{diag}(\lambda_1^2,\lambda_2^2)\); see Equation 2.12). However, overlaying the ellipse on the CoG trajectory is useful for verifying that the procedure has been implemented correctly: if so, the ellipse should provide a reasonable fit to the data.

We begin by creating an ellipse, scaling it according to the eigenvalues computed in the previous section, and plotting it:

Show code 
plot(x_cog, y_cog,

y <- sin(
  seq(0, 2 * pi, length.out = 100)
  ) # y coordinates of the ellipse

x <- x * lambda[1] * 1.96 # scaling according to the first eigenvalue
y <- y * lambda[2] * 1.96 # scaling according to the second eigenvalue

lines(x, y, 
  type = "l", 
  col = "pink",

Although the ellipse appears to have been scaled correctly, it does not fit the data well. This occurs because the ellipse was drawn in the original \((x,y)\) reference system. We must therefore rotate it into the reference system defined by the principal axes, \((x',y')\). This can be done in R using the matrix multiplication operator (%*%):

Show code 
xpyp <- cbind(x,y) %*% 
  t(eigvec)      # transforming between reference systems
xp <- xpyp[ , 1] # all elements of the first column of the transformed data
yp <- xpyp[ , 2] # all elements of the second column of the transformed data

# Then just rerun the commands above
plot(x_cog, y_cog, 
  type = "l", 
  col = "turquoise4", 
  asp = 1, 
  main = "CoG Statokinesigram", 
  xlab = "medial-lateral (cm)", 
  ylab = "Anterior-Posterior (cm)"
  )

abline(h = 0, lty = "dotted") # adding HORIZONTAL line at 0 cm
abline(v = 0, lty = "dotted") # adding VERTICAL line at 0 cm

lines(xp, yp, 
      type = "l", 
      col = "pink", 
      lwd = 8
      ) # drawing ellipse

legend(
  "bottomleft", 
  inset = 0.02, 
  c("CoG","Ellipse (1.96\U03BB)"), 
  col = c("turquoise4","pink"), 
  lty = 1, 
  lwd = c(2, 8), 
  cex = 0.8
  )

We changed reference system by multiplying the \(x\) and \(y\) coordinates of the ellipse (cbind(x, y)) by the transpose (t()) of the eigvec matrix, which contains the two eigenvectors, one in each column.


Computing the CoG Elliptic Area

After identifying the axes of the ellipse that best fits the CoG data (see Preliminaries on the Elliptic Area), we compute the elliptic area \(EA\) using Equation 2.12, restated here:

\[ EA = \pi k^2 \sqrt{\lambda_1 \lambda_2} \]

where \(\sqrt{\lambda_1}\) and \(\sqrt{\lambda_2}\) are the standard deviations of the CoG coordinates in the reference system aligned with the ellipse semi-axes (see Figure 7.11), and \(k\) is the scaling factor determining the proportion of data points included in the ellipse. For \(k=1.96\), the ellipse encloses approximately 85% of the CoG samples.

For the data shown in Figure 7.11, the CoG elliptic area is EA = \(7.8 \textrm{ cm}^2\):

EA = pi * 1.96^2 * prod(lambda)

Mean Speed and Standard Deviation

Computing these stabilometric metrics simply requires translating the equations introduced in Section 2.2.5.1 into code. They are restated below for convenience.

Several procedures could be used to compute these two metrics. Both require processing all \(N=3000\) CoG samples, corresponding to the \(30\textrm{ s}\) processing window defined when Trimming Sway Data. Since loops in R will only be introduced in Chapter 8, we will use a different approach here.

NoteDo it yourself

Unlike the Elliptic Area, no code will be provided for computing these two metrics. This task is left as an exercise at the end of this chapter.


Mean CoG Speed (\(v_{x_\textrm{cog}}\) and \(v_{y_\textrm{cog}}\))

We may either:

  • compute the total CoG displacement and divide it by the duration of the processing epoch, \(\frac{N-1}{sf}=29.99\textrm{ s}\);

  • compute the \(N-1=2999\) instantaneous speed values from each pair of consecutive CoG samples and then average them.

Both approaches lead to the following equation:

\[ v_x=\frac{sf}{(N-1)}\sum_{i = 1}^{N-1} |x_\textrm{cog}(i+1)-x_\textrm{cog}(i)| \]

where \(x_\textrm{cog}\) denotes the CoG coordinate in the frontal plane8 (medio-lateral direction), sampled at frequency \(sf\) for a total of \(N\) samples.


CoG Standard Deviation (\(\sigma_{x_\textrm{cog}}\) and \(\sigma_{y_\textrm{cog}}\))

The CoG Standard Deviation is defined as:

\[ \sigma_x=\sqrt{\frac{1}{N}\sum_{i = 1}^{N} \left(x_\textrm{cog}(i)-\bar{x}_\textrm{cog}\right)^2} \]

where \(x_\textrm{cog}\) and \(\bar{x}_\textrm{cog}\) denote, respectively, the CoG coordinate and the mean CoG position in the frontal plane9 (medio-lateral direction).

Notice that \(\bar{x}_\textrm{cog}=0\) because of the Demeaning Sway Data procedure described above. Therefore, in this case, the Standard Deviation and Root Mean Square values coincide.

Note\(N\) or \(N-1\)?

Some authors use a slightly different definition of standard deviation, with \(N-1\) instead of \(N\) in the denominator.

The reason for this distinction will become clear later in the course. Because the number of CoG samples is typically large, the two definitions can be considered practically equivalent.
