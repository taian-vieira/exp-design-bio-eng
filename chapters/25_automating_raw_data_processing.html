In Chapter 7, we saw how to use R to compute metrics for assessing postural sway. From smartphone orientation data, we computed the CoG Elliptic Area, Mean CoG Speed, and CoG Standard Deviation.

Briefly, this chapter covers:

  • Custom Functions in R, making code simpler and more readable with custom functions.

8.1 Organizing CoG Raw Data

Data organization is as important as any other step in a project. By data organization, we do not mean only where files are stored, but also how the project is structured: how files are named, how folders are arranged, and how scripts refer to the data. To illustrate the importance of data organization, we will use the structure of the shared files as an example.

Inside the compressed folder that can be downloaded from the link above, you should find two folders:

  • data, which contains two subfolders:

    • s_020966, containing smartphone orientation data for the participant with registration number 020966;

    • s_111370, containing smartphone orientation data for the participant with registration number 111370.

      Each of these folders contains four xls files. Their names explicitly indicate the experimental condition under which postural sway data were collected:

      • both legs, with eyes open;
      • both legs, with eyes closed;
      • single leg, with eyes open;
      • single leg, with eyes closed.

  • scripts, where the R files used to process the raw data and analyze the processed data will be stored.

After downloading these two folders, place both inside any parent directory you prefer. We will refer to this parent directory as the project directory. On my computer, as illustrated in Figure 8.1, the project directory is named exp_design_bio_eng.

The name of the project directory is not strictly important, as long as it is informative to you and to the colleagues working with you on the project.

For example, you may often work on the same project as other colleagues. Even if you use a cloud storage or version control service to track changes across project files, the code written to process and analyze the data will likely run locally on your own machine. To ensure that your scripts run successfully on different computers, it is essential to use relative paths.


Relative Paths with an R Project

Absolute paths uniquely identify the location of files or directories on a computer. Because they depend on the directory structure of a particular machine, they differ from one computer to another. After downloading the shared data, the absolute path to the data folder on your computer will therefore differ from the corresponding path on mine. Consequently, any scripts you write and place in the scripts folder may fail when run on a different computer because the absolute paths stored in the scripts will no longer be valid (or even on your own computer if the project is moved to another location in the future).

Ideally, the scripts we write should reference files and folders using paths that are relative to the project directory. We can achieve this by creating an R Project. To do so, click File → New Project… in the RStudio menu. This opens the New Project Wizard, shown in Figure 8.2.

Select Existing Directory and then choose the project directory containing the data and scripts folders. RStudio will create an .Rproj file, named after the project directory, inside that directory (see Figures 8.1 and 8.3).

We will not discuss the details of an R Project here1. The important point is that whenever you open the .Rproj file in RStudio, the corresponding project directory is automatically set as the current Working Directory (see the Files section in Chapter 5). You can verify this by querying the current Working Directory with the getwd() function.

getwd()
[1] "r_studio/tutorial/exp_design_bio_eng"

The last directory displayed should correspond to your project directory containing the data and scripts folders. In my case, this directory is exp_design_bio_eng.

8.2 Looping over Directories and Files

Given the organization of the CoG raw data shown in Figure 8.1, we can use nested loops2 to process one file at a time for each participant.


Nesting for Loops

In our case, two loops are required. The outer loop iterates over the participants—that is, over the folders s_020966 and s_111370. The inner loop iterates over the four xls files contained within each participant’s folder. One possible implementation is:

# storing the folders of the two individuals in an R variable
sub_folders <- c("s_020966", "s_111370") # subjects' folders
# storing the four xls files into an R variable
xls_files <- c("both_legs_eyes_closed.xls",
              "both_legs_eyes_open.xls",
              "single_leg_eyes_closed.xls",
              "single_leg_eyes_open.xls")

# outer loop (moving across subjects)
for (i in 1:length(sub_folders)){
  # inner loop (moving across xls files)
  for (k in 1:length(xls_files)){
    # here we have access to the i-th subject folder and the k-th xls file
    
    cat("i = ", i, ", k = ", k, ", subject folder: ", sub_folders[i], ", xls file: ", xls_files[k], "\n", sep = "")
  }
}

We first define two variables, sub_folders and xls_files, which store the names of the participant folders (see Figure 8.1) and the xls files contained in each folder, respectively.

Next, we create two for loops using the syntax:

for (sequence to loop over){
  statements
}

Inside the inner loop, we use the cat() function to display the values of the iteration variables i and k, together with the corresponding elements of sub_folders and xls_files.

This approach is not ideal because it requires manually specifying the names of all participant folders and files. Moreover, not every participant necessarily has four xls files. For example, data acquisition may have been interrupted or a participant may have withdrawn before completing all experimental conditions.

A more robust approach is to obtain the folder and file names directly using the list.dirs() and list.files() functions, provided that the data are organized as described in Organizing CoG Raw Data (see Figure 8.1).

sub_folders <- list.dirs("data", recursive = FALSE, full.names = FALSE) # directories inside the project folder

for (i in sub_folders){ # outer loop
  xls_files <- list.files(file.path("data", i)) # i is not a number, but an element of sub_folders variable
  
  for (k in xls_files){ # inner loop
    cat("subject folder: ", i, ", xls file: ", k,  "\n", sep = "")
  }
}

In this version, the syntax of the for loop changes slightly. The iteration variables i and k no longer represent numeric indices; instead, they directly store the elements of sub_folders and xls_files, respectively. This makes the code simpler and easier to read. Moreover, if a participant folder contains fewer or more than four xls files, every available file will still be processed correctly3.

CHALLENGE

Try to avoid using file.path() inside list.files() to retrieve the files contained in the i-th participant folder.


Initializing the Output Data Frame

The output data frame should follow the structure shown in Table 8.1. There are two common approaches:

  • initialize an empty data frame and add one row at a time while looping over participant folders and files;
  • initialize a sufficiently large data frame containing placeholder values (NA, 0, "", etc.), update its rows as the files are processed, and remove any unused rows at the end.

The second approach is generally more efficient because it preallocates memory for the data frame, potentially saving hundreds of milliseconds of execution time (see this post). Although the exact number of observations may not be known a priori, we can usually define a reasonable upper bound, n, for the number of observations:

n <- length(sub_folders) * 5 # total number of expected observations
n
[1] 10

length(sub_folders) corresponds to the number of participant folders to be processed, which in our case is 2. Since we expect four xls files in each folder, multiplying the number of participant folders by five, rather than four, ensures that the initialized data frame is sufficiently large. In our case, this corresponds to n = 10 rows.

We can now initialize the data frame:

df_cog <- data.frame(
  subject = character(n), 
  eyes = factor(character(n), levels = c("open", "closed")), 
  standing_on = factor(character(n), levels = c("single leg", "both legs")),
  a_cog = double(n),    # elliptic area
  v_x = double(n),      # mean speed in x direction
  v_y = double(n),      # mean speed in y direction
  sigma_x = double(n),  # st. deviation in x direction
  sigma_y = double(n)   # st. deviation in y direction
)

Inside the inner loop, we can now populate df_cog as each file is processed. For every observation, we must determine the values of the eight variables (columns) listed in Table 8.1. We begin by extracting information about the participant and the experimental condition.


Retrieving Participant IDs and Test Conditions

The first column of df_cog should contain the registration number of each participant from whom smartphone orientation data were collected. Columns 2 and 3 should describe the experimental conditions, namely whether the participant performed the test with eyes open or closed, and whether they stood on one or both legs.

In many research projects, this information is stored in a README file or in separate files containing metadata. In our example, however, the required metadata can be extracted directly from the names of the participant folders (sub_folders) and the corresponding xls files.

ImportantRespecting naming conventions

For the approach presented here to work, folders and files must follow a consistent naming convention (see Section 8.1).

The experimental protocol considered for collecting postural sways data, for example, does not impose any convention for the name of files. However, it requires that the base of support and the standing condition are set appropriately in the SmartPosture experiment (see Section 3.5)

Regardless of the convention adopted in a study, following it consistently greatly facilitates data processing.

There are two common approaches for extracting the required metadata. The first relies on the if-else statement:

if (expression to be tested){
some statements
} else {
other statements
}

for (i in sub_folders){
  xls_files <- list.files(file.path("data", i)) 
  subject <- sub(".*_","",i) # taking all characters after  underscore

  for (k in xls_files){
    
    # is the current trial eyes open or eyes closed?
    if (grepl("open", k)){
      eyes <- "open"
    } else {
      eyes <- "closed"
    }
    
    # is the current trial single leg or both legs?
    if (grepl("single", k)){
      standing_on <- "single leg"
    } else {
      standing_on <- "both legs"
    }

    cat("subject folder: ", i, ", xls file: ", k, ",\nsubject: ", subject, ", standing on: ", standing_on, ", eyes: ", eyes, "\n\n", sep = "")
  }
}

The grepl() function checks whether the strings "open" and "single" appear in the name of the current xls file. Depending on the outcome of these tests, the variables eyes and standing_on are assigned the appropriate values.

The participant ID is extracted from the folder name by retaining all characters following the final underscore ("_") using the sub() function4.

eyes <- c("closed", "open") # possible entries
standing_on <- c("both legs", "single leg") # possible entries

for (i in sub_folders){
  xls_files <- list.files(file.path("data", i)) 
  
  for (k in xls_files){
    
    info_required <- c(
      sub(".*_","",i), # subject code (character after last underscore)
      eyes[grepl("open", k)+1], # eyes open OR eyes closed
      standing_on[grepl("single", k)+1] # single OR both legs
    )
    
    print(info_required)
  }
}

A cleaner alternative is to use the logical output of grepl() directly as an index. This approach eliminates the if-else blocks altogether.

Here, we use the logical value returned by grepl() to index the vectors eyes and standing_on. Since logical values are coerced to integers when used in arithmetic (FALSE = 0, TRUE = 1), adding 1 converts them into valid R indices (1 or 2).

Both approaches produce the same result. The choice is largely a matter of style, although the indexing approach results in shorter and arguably more readable code. Using this syntax, we can now populate the first three columns of df_cog with the participant ID, visual condition, and standing condition.

count <- 0 # used to index df_cog

for (i in sub_folders){
  xls_files <- list.files(file.path("data", i)) 
  
  for (k in xls_files){
    count <- count + 1 # update count
    df_cog[count, 1:3] <- c(
      sub(".*_","",i), # subject code (character after last underscore)
      c("closed", "open")[grepl("open", k)+1], # eyes open OR eyes closed
      c("both legs", "single leg")[grepl("single", k)+1] # single OR both legs
    )
    
  }
}

8.3 R Script (main)

In the previous section, we populated the first three columns of the df_cog data frame shown in Table 8.1. We must now complete the remaining columns by computing the CoG metrics selected to characterize postural control.

The procedures required to compute these variables from the smartphone orientation data were presented in detail in Chapter 7. Inside the nested loops, we therefore need to perform the following processing steps:

  • Pre-processing the sway data
    • Trimming: removing the initial transient and standardizing the number of CoG samples;
    • Demeaning: removing the mean CoG position, which is not of practical interest.
  • Computing the CoG metrics
    • Elliptic Area;
    • Mean CoG Speed in the frontal and sagittal planes;
    • CoG Standard Deviation in the frontal and sagittal planes.

So far, we have entered our code directly in the R Console. We will now create an R script, where all processing steps required to compute the CoG variables will be stored.

An R script is simply a plain text file containing R code. There are two common ways to create one:

  • from File → New File → R Script (see Figure 8.4);
  • or by clicking the New File icon in the upper-left corner of the RStudio window (see Figure 8.5).

Either method creates a blank script named Untitled1. Save this file as main.R inside the scripts directory (see Figure 8.1).

The script developed below retrieves the participant metadata (registration number and test conditions) and computes the CoG Elliptic Area for each xls file.

library(readxl) # for reading xls files

dur <- 30  # (s) duration considered for processing
sub_folders <- list.dirs("data", recursive = FALSE, full.names = FALSE) # directories inside the project folder
n <- length(sub_folders) * 5 # number of expected observations

# initializing data frame with all columns from Table 5.1
df_cog <- data.frame(
  subject = character(n), 
  eyes = factor(character(n), levels = c("open", "closed")), 
  standing_on = factor(character(n), levels = c("single leg", "both legs")),
  a_cog = double(n),    # elliptic area
  v_x = double(n),      # mean CoG speed in x direction
  v_y = double(n),      # mean CoG speed in y direction
  sigma_x = double(n),  # CoG st. deviation in x direction
  sigma_y = double(n)   # CoG st. deviation in y direction
)

count <- 0 # used to index df_cog

for (i in sub_folders){ # outer loop (over subjects' folders)
  xls_files <- list.files(file.path("data", i)) # retrieving xls files

  for (k in xls_files){ # inner loop (over xls files)
    count <- count + 1 # update count
    
    # SUBJECT AND TEST CONDITIONS - updating df_cog with metadata
    df_cog$subject[count] <- sub(".*_","",i) # subject code (character after last underscore)
    df_cog$eyes[count] <- c("closed", "open")[grepl("open", k)+1] # eyes open OR eyes closed
    df_cog$standing_on[count] <- c("both legs", "single leg")[grepl("single", k)+1] # single or both legs
    
    # loading raw data
    df <- read_xls(file.path("data", i, k)) # data frame with smartphone orientation data

    # COMPUTING COG VARIABLES, PART 1: PRE-PROCESSING STEPS

    # trimming CoG coordinates
    si <- diff(df$time)          # all sampling intervals
    sf <- round(1/mean(si))      # mean sampling frequency (100 Hz)
    y_cog <- head(df$y_cog[df$time>5], dur * sf) # 30 s of CoG samples in anterior-posterior (Y) direction
    x_cog <- head(df$x_cog[df$time>5], dur * sf) # 30 s of CoG samples in medial-lateral (X) direction

    # demeaning CoG coordinates
    y_cog <- y_cog - mean(y_cog, na.rm = TRUE) # Y direction
    x_cog <- x_cog - mean(x_cog, na.rm = TRUE) # X direction

    # COMPUTING COG VARIABLES, PART 2: PROCESSING STEPS

    # computing the elliptic area and updating df_cog
    xy <- cbind(x_cog, y_cog) # matrix with N rows (sf * dur) and 2 cols
    head(xy, 5)              # shows the first 5 rows of the xy matrix
    E <- cov(xy)              # covariance matrix
    xy_prime <- eigen(E)  # computing eigenvalues and eigenvectors
    lambda <- sqrt(xy_prime$values) # square root of eigen values
    eigvec <- xy_prime$vectors      # eigen vectors
    df_cog$a_cog[count] <- pi * 1.96^2 * prod(lambda) # elliptic area (containing 85% of CoP samples)

  }
}

df_cog <- df_cog[1:count, ] # keeping only count observations

8.4 Creating R Functions

Code should be as clean, concise, and readable as possible. This makes it easier both for others to understand and maintain your code and for you to revisit it in the future. Numerous style guides for R programming are available; one example is this guide.

Rather than allowing main.R to become long and difficult to read, we will move the computation of the CoG variables into a separate R function.

The general syntax for defining a function in R is:

function_name <- function(parameter_1, parameter_2, ...) {
  statements
  return(value_to_be_returned)
}

There are two common ways in which an R function can update df_cog: by value or by reference. The later approach is discouraged (see Tip 8.1).


Updating CoG Metrics by Value

With this approach, the function computes the desired metrics from the raw data and returns them to the calling code. The returned values are then assigned to the appropriate row of df_cog.

To do this, we move the portion of code in main.R responsible for computing the CoG metrics into a new R script. Save this script as cog_functions.R inside the functions folder (see Figure 8.1).

compute_cog_metrics <- function (df_raw_data) {

  # COMPUTING COG VARIABLES, PART 1: PRE-PROCESSING STEPS
  # trimming CoG coordinates
  
  si <- diff(df_raw_data$time)          # all sampling intervals
  sf <- round(1/mean(si))      # mean sampling frequency (100 Hz)
  y_cog <- head(df_raw_data$y_cog[df$time>5], dur * sf) # 30 s of pitch sample
  x_cog <- head(df_raw_data$x_cog[df$time>5], dur * sf) # 30 s of yaw sample
  
  # demeaning CoG coordinates
  y_cog <- y_cog - mean(y_cog, na.rm = TRUE) # demeaned pitch
  x_cog <- x_cog - mean(x_cog, na.rm = TRUE) # demeaned yaw
  
  # COMPUTING COG VARIABLES, PART 2: PROCESSING STEPS
  
  # computing the elliptic area and updating df_cog
  xy = cbind(x_cog, y_cog) # matrix with N rows (sf * dur) and 2 cols
  head(xy, 5)              # shows the first 5 rows of the xy matrix
  E <- cov(xy)              # covariance matrix
  xy_prime <- eigen(E)  # computing eigenvalues and eigenvectors
  lambda <- sqrt(xy_prime$values) # square root of eigen values
  eigvec <- xy_prime$vectors      # eigen vectors
  a_cog <- pi * 1.96^2 * prod(lambda) # elliptic area
  
  return(a_cog)
}

Examining the function reveals that:

  • the CoG metrics are computed by the function compute_cog_metrics();
  • the function has a single input parameter (df_raw_data), whose argument should be a data frame containing the raw CoG data;
  • the function contains essentially the same code previously included in main.R, with one important addition: the statement return(a_cog), which returns the computed CoG Elliptic Area.

Before this function can be used, the R script containing it must be sourced. Once this has been done, compute_cog_metrics() becomes available in the current R session and should appear in the Environment pane (or can be listed with the ls() function).
